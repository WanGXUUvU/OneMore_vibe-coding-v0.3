# 工作流模式详解

## 完整工作流（默认）

```
读取仓库状态
    ↓
[需要 TASK-000?] ──是──→ 创建 TASK-000 → Brainstorm Review 关卡
    ↓否                                         ↓
确认范围 ←───────────────────────────────────────┘
    ↓
选择车道 → Fast / Standard / Strict
    ↓
执行实施（按车道流程）
    ↓
Verify Minimum（验证最低标准）
    ↓
Review Minimum（评审最低标准）
    ↓
Sync Review 关卡（停止等确认）
```

## Fast 车道流程

```
确认范围
    ↓
generator（编写代码）
    ↓
evaluator（验证结果）
    ↓
Verify + Review
    ↓
Sync Review 关卡
```

**适用场景：**
- 修 bug
- 改配置文件
- 加日志/注释
- 文档更新
- 单文件小改动

**时间预估：** 5-15 分钟

## Standard 车道流程

```
确认范围
    ↓
planner（制定计划）
    ↓
Plan Review 关卡（停止等确认）
    ↓
generator（按计划编写代码）
    ↓
evaluator（验证结果）
    ↓
Verify + Review
    ↓
Sync Review 关卡
```

**适用场景：**
- 新增功能页面
- 重构模块
- 跨多文件改动
- 新增 API 接口

**时间预估：** 15-60 分钟

## Strict 车道流程

```
确认范围
    ↓
planner（制定计划）
    ↓
Plan Review 关卡（停止等确认）
    ↓
generator（按计划编写代码）
    ↓
evaluator（首次验证）
    ↓
fixer（修复发现的问题）
    ↓
evaluator（二次验证）
    ↓
Verify + Review
    ↓
Sync Review 关卡
```

**适用场景：**
- 数据库结构变更
- 删除表/字段
- 上线前变更
- 安全相关修改
- 不可逆操作

**时间预估：** 30-120 分钟

## 单代理串行回退

当子代理不可用时，由主代理手动切换角色：

```
同一个代理按顺序扮演：
1. planner 角色 → 输出计划
2. [等待用户确认计划]
3. generator 角色 → 输出代码
4. evaluator 角色 → 输出验证结果
5. [如果是 Strict 车道]
6. fixer 角色 → 输出修复
7. evaluator 角色 → 输出二次验证
```

**关键原则：** 即使是同一个代理，也要在不同角色间切换，不能在 generator 阶段跳过 evaluator 直接宣告完成。

## 任务卡生命周期

```
创建 → 确认范围 → 选择车道 → 执行 → 验证 → 评审 → 关闭
  ↑                                        ↓
  └────── 验证失败，返回修改 ←──────────────┘
```

## 关卡决策树

### Brainstorm Review 决策

```
TASK-000 是否可执行？
├── 是 → 用户确认 → 创建 TASK-001，进入实施
├── 否 → 目标不清晰 → 完善"项目名称+目标"
│       现状不清晰 → 完善"现状与背景"
│       角色不清晰 → 完善"用户角色"
│       约束不清晰 → 完善"技术约束"
│       范围不清晰 → 完善"Out of Scope"
│       验收不清晰 → 完善"Done when"
└── 不确定 → 默认认为不清晰，继续完善
```

### Plan Review 决策

```
计划是否合理？
├── 是 → 用户确认 → 开始 generator
├── 否 → 遗漏步骤 → 补充计划
│       步骤过多 → 拆分为多个 TASK
│       风险未识别 → 添加风险说明
└── 不确定 → 默认认为需要修改
```

### Sync Review 决策

```
变更是否可安全同步？
├── 是 → 用户确认 → 更新 STATUS.md，任务关闭
├── 否 → 有阻断问题 → 返回 fixer
│       风险过高 → 降级处理或回退
└── 不确定 → 默认认为需要用户确认
```

## 异常处理

### 验证失败

1. 记录失败原因
2. 如果是 Fast 车道 → 升级为 Standard 车道
3. 如果是 Standard 车道 → 根据严重程度决定是否升级为 Strict
4. 如果是 Strict 车道 → fixer 修复后重新验证

### 范围蔓延

1. 发现新需求时，不在当前任务中实现
2. 创建新的 PATCH-TASK 或 TASK-xxx 记录新需求
3. 当前任务按原范围完成

### 用户中断

1. 保存当前进度到 STATUS.md
2. 记录当前关卡和下一步行动
3. 下次触发技能时从断点继续
